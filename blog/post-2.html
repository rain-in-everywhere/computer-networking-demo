理解 CDN：为什么"距离"是互联网上最昂贵的资源

内容分发网络如何通过"把内容搬到用户身边"来消除光速延迟

2026-06-27

光在光纤中的传播速度约为 200,000 km/s（真空的 2/3）。从北京到纽约约 11,000 km，光速单程就需要 55ms。加上路由跳转、排队延迟，跨太平洋的 RTT 通常高达 150-300ms——这就是 CDN 存在的根本原因。

没有 CDN 的世界

用户（东京）请求 example.com
↓ （DNS 解析 → 美国源站 IP）
TCP + TLS 握手：东京 ↔ 美国 × 3 次往返
↓ RTT ≈ 120ms，握手总耗时 ≈ 360ms
HTTP 请求 → 等待响应 → 总 TTFB ≈ 480ms+

结论：每个远距离用户都在为"光速"付出代价

有了 CDN 的世界

用户（东京）请求 cdn.example.com
↓ （DNS Anycast → 最近的 CDN 节点：东京/NRT）
TCP + TLS 握手：东京 ↔ 东京
↓ RTT ≈ 5ms，握手总耗时 ≈ 15ms
HTTP 请求 → 边缘缓存命中 → TTFB ≈ 20ms

结论：CDN 把响应延迟降低了一个数量级

Cloudflare 的 CDN 架构

Cloudflare 在全球拥有 330+ 个边缘数据中心，每个节点都能：

  • 缓存静态资源（HTML、CSS、JS、图片）在边缘
  • 执行 Serverless 代码（Cloudflare Workers）在边缘
  • SSL 终端：在边缘节点完成 TLS 握手
  • DDoS 防护：在边缘识别并阻断攻击流量

Anycast：让一个 IP 出现在世界各地

CDN 依赖 Anycast 技术：同一个 IP 地址同时宣告在全球多个地点。当用户访问这个 IP 时，BGP 路由协议会自动选择跳数最少、延迟最低的路径，将流量导到最近的节点。

IP 1.1.1.1 同时在：
─ 洛杉矶（LAX） 宣告
─ 法兰克福（FRA）宣告
─ 东京（NRT） 宣告
─ 新加坡（SIN） 宣告
……全球 330+ 个节点

用户的路由器自动选择最近的路径 → Anycast 实现了天然的地理负载均衡

动手验证

访问本网站的 CDN 演示页面，你会看到：

  • colo：本次请求命中哪个 Cloudflare 边缘数据中心
  • httpProtocol：使用的 HTTP 协议版本（HTTP/2 或 HTTP/3）
  • tlsVersion / tlsCipher：TLS 加密参数
  • asn：你的 ISP 的自治系统编号

这些信息来自 Cloudflare 注入的 request.cf 对象——它让你在 Worker 代码中直接读取请求的边缘元数据。

总结

CDN 核心思想：把数据搬到离用户更近的地方
─ 减少物理距离 → 降低光速延迟
─ Anycast 路由 → 自动选择最近节点
─ 边缘计算 → 不仅缓存，还能执行代码
─ SSL 终端在边缘 → 减少 TLS 握手延迟

本网站本身就是 CDN 的实践案例：
所有内容通过 Cloudflare Workers 在全球 330+ 节点上运行
