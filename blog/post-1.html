数据包的网络之旅

从浏览器输入 URL 到页面呈现，网络协议栈的完整协作过程

2026-06-28

当你打开浏览器，输入 https://example.com 并按下回车，看似一瞬间的事情，背后却发生了数十个网络协议的协作。这篇文章追踪一个数据包的完整旅程。

第一步：DNS 解析 — 把域名翻译成 IP 地址

计算机不认识 "example.com" 这个名字，它只认识 IP 地址（如 93.184.216.34）。DNS（Domain Name System）就是互联网的电话簿。

① 浏览器检查本地 DNS 缓存
② 没命中 → 查询操作系统缓存（hosts 文件）
③ 没命中 → 向递归 DNS 解析器（如 1.1.1.1）发起查询
④ 解析器依次查询：根域名服务器 → TLD 服务器（.com）→ 权威 DNS 服务器
⑤ 返回 A 记录：example.com → 93.184.216.34
⑥ 浏览器拿到 IP，DNS 解析完成（约 20-120ms）

如今，DNS 查询可以通过 DNS-over-HTTPS (DoH) 加密传输，防止运营商或中间人窃取你的浏览记录。本网站的 DNS 演示页面 就使用了 DoH 协议。

第二步：TCP 三次握手 — 建立可靠连接

拿到 IP 地址后，浏览器通过 TCP（传输控制协议）与服务器建立连接。TCP 是面向连接、可靠的传输协议。

① 客户端 → SYN (seq=x) → 服务器
② 客户端 ← SYN+ACK (seq=y, ack=x+1) ← 服务器
③ 客户端 → ACK (ack=y+1) → 服务器
握手完成，连接建立！（约 1 个 RTT）

第三步：TLS 握手 — 加密通信

因为是 HTTPS，TCP 连接建立后需要进行 TLS 握手来协商加密参数。TLS 1.3 只需 1-RTT：

① ClientHello：客户端发送支持的加密套件 + 密钥共享（DH）
② ServerHello：服务器选择加密套件 + 证书 + 密钥共享
③ 双方计算会话密钥，后续通信使用对称加密
TLS 1.3 握手完成（+1 RTT）

你可以访问本网站的 CDN 演示页面，查看你的连接使用的 TLS 版本和加密套件。

第四步：HTTP 请求 — 获取资源

加密通道建立后，浏览器发送 HTTP 请求。如果是 HTTP/2，一个 TCP 连接可以多路复用多个请求：

GET / HTTP/2
Host: example.com
Accept: text/html
Accept-Encoding: gzip, br
User-Agent: Mozilla/5.0...

第五步：CDN 边缘响应

如果网站部署了 CDN（如 Cloudflare），请求不会到达源站，而是由离用户最近的边缘节点直接响应：

用户（上海）→ Cloudflare 边缘节点（上海/SHA）→ 缓存命中 → 直接返回
延迟仅 5-20ms，而非跨太平洋的 150ms+

这就是本网站的核心架构——所有内容都通过 Cloudflare Workers 在边缘节点生成和缓存。

总结：完整的时间线

DNS 解析： 20-120ms
TCP 握手（1-RTT）： 取决于距离（国内 20-50ms）
TLS 握手（1-RTT）： +1 RTT（TLS 1.3）
HTTP 请求： 1 RTT
总计首字节时间（TTFB）：通常 100-300ms

——如果使用 CDN 边缘缓存——
TTFB 可能降至 10-50ms

你可以在浏览器中打开 DevTools → Network 面板，刷新这个页面，观察每个资源的 Timing 瀑布图，验证上面的每一阶段。
