⚡ HTTP 缓存策略

通过 Cache-Control 和 ETag 头控制浏览器缓存行为，大幅减少网络请求

📖 HTTP 缓存的工作原理

HTTP 缓存通过响应头告诉浏览器：这个资源可以缓存多久、何时需要重新验证。 合理使用缓存可以减少 70-90% 的 HTTP 请求，显著提升页面加载速度。

指令含义浏览器行为
no-store禁止缓存每次都向服务器发请求
no-cache缓存但需验证每次使用前发条件请求
max-age=N缓存 N 秒N 秒内直接用缓存
ETag资源版本标识条件请求（If-None-Match）→ 304
immutable永远不变即使刷新页面也不请求

🧪 实验：对比缓存策略

选择一种缓存策略，多次点击"发送请求"。打开 DevTools Network 面板， 观察 Status 列（200 → 304 → "from disk cache"）和 Size 列的变化。

💡 知识点

304 Not Modified

服务器告诉浏览器"资源没变，用你的缓存副本"。响应体为空， 节省了传输带宽。适合 ETag 和 Last-Modified 验证。

from disk cache

浏览器直接从磁盘缓存读取，完全不发网络请求。 只有 max-age 未过期或 immutable 资源才会这样。

immutable 模式

现代前端构建工具（Vite/Webpack）给文件加 content hash， 如 app.abc123.js，配合 immutable 实现永久缓存。

实践建议

HTML：短缓存（max-age=0 或 no-cache）
CSS/JS（带 hash）：immutable + 长期缓存
图片/字体：max-age=1 年
API 响应：根据需要，通常 no-store
